# Hauler's Dream — Mod Compatibility Feature-Risk Map

Developer reference. A per-feature compatibility risk map for the mods Hauler's Dream (HD) is built to coexist with, produced by a multi-agent investigation: one investigator per feature family read the mod's **source** (cloned, or decompiled for closed-source) and its **bug-fix history**, traced every feature against HD's patch surface, then an adversarial skeptic verified each family's load-bearing claims, then a synthesis pass produced the plan at the end. For each mod feature: the vanilla mechanism it uses, which HD subsystem overlaps, the risk level, and the recommended fix — preferring a **general pattern** (covers any mod with that feature) over a mod-specific patch. See `COMPATIBILITY.md` for the user-facing summary.

**Risk legend:** `HIGH` = can break a feature / loop / strand items · `MEDIUM` = wasted work or a transient over-estimate · `LOW` = cosmetic or rare edge · `NONE` = no overlap.

## Family: `storage`

HD is structurally well-defended against this storage family because every hot path validates candidate cells exclusively through vanilla seams (IsGoodStoreCell→NoStorageBlockersIn/GetMaxItemsAllowedInCell, CanStackWith, TryFindBestBetterStorageFor, Accepts) that all three mods patch — so modded acceptance/capacity/routing apply automatically, and the deposit-time re-gate (FindTargetOrDrop + PlaceHauledThingInCell IsGoodStoreCell re-validation) is the self-correcting authority that bounds any over-estimate to one-cycle churn rather than loss. Adaptive Storage Framework and LWM Deep Storage keep stored items SPAWNED on-map (in thingGrid / ListerThings.ThingsOfDef), so HD's sweep/route/stack/count features see and handle them correctly; Storage Network keeps items DESPAWNED inside server/terminal ThingOwners, so HD's ThingsOfDef-based features simply don't fire over network-only inventory (graceful degrade, handled by HD's opt-in SN bulk-load shim + SN's own ItemAvailability postfix and StartJob auto-spawn). The two genuinely live risks are: (1) HaulToStack's no-cell-reservation prefix racing multiple pawns onto the same LWM/ASF multi-stack cell, where LWM's added storing-time DELAY on PlaceHauledThingInCell widens the race window vs vanilla's instant place (the LWM author himself flags 'Haul To Stack? conflicting reports' and warns its toil chain 'could break' when moving many things at once) — still bounded by deposit re-gate, but more wasted trips; and (2) HD's StorageSpaceForDef calling the STANDALONE GridsUtility.GetItemStackSpaceLeftFor (which LWM does NOT patch — only the HaulToCellStorageJob-internal call is transpiled), so for LWM mass/weight-limited DSUs the per-cell space is over-estimated (count-only, mass-blind) — HD documents this as a safe upper bound that self-corrects at deposit. Both reduce to extra churn, never cargo loss or storage-mod-rule bypass. No mod-specific patch is required; the general pattern (validate through vanilla APIs; treat non-spawned container contents as out-of-reach; let the deposit re-gate be authority) already covers all three and any future Building_Storage-derived mod.

### Prior bugs learned from (their bug-fix history)

- n/a to HD — **Weird cache bug making pawns haul ZERO items to storage** _(LWM Deep Storage git commits 5449c23 / 1b6c8e0 ('Fix for weird cache bug making pawns haul zero items to storage'))_ — LWM's MapComponentDS capacity cache returned zero free space when stale, so pawns wouldn't haul. HD keeps NO DSU/per-cell capacity cache of its own — it reads live via IsGoodStoreCell + GetItemStackSpaceLeftFor every probe (HD's only caches are per-tick plan/work memos keyed on game state, re-validated by IsGoodStoreCell on hit). So HD cannot reintroduce a stale-capacity-zero bug. The lesson reinforces: never cache modded storage capacity; always re-read through the vanilla seam — which HD already does.
- n/a to HD — **Cache doesn't clear if the last item in a unit isn't a full stack but is filled (partial-stack capacity miscount)** _(LWM Deep Storage git commit 41f357d ('Fix bug where cache doesn't clear if last item in unit isn't full stack but is filled'))_ — A partial-but-present stack in the last slot left the cache thinking the cell had room. This is exactly the partial-stack reasoning HD's HaulToStack does (CellHasPartialStackOf: other.stackCount < stackLimit && CanStackWith). HD does NOT cache this — it reads thingGrid.ThingsListAtFast live each call and re-gates with IsGoodStoreCell (which runs LWM's StackableAt). So HD's partial-stack top-up reflects LWM's live capacity, not a stale snapshot. Class of bug avoided by construction.
- n/a to HD — **Gizmo patch crashing on 1.5 caused by IL changes; multiple 'try statements to keep things running better'** _(LWM Deep Storage commits d468e83, c4f9c25)_ — LWM's own transpiler/UI fragility across versions — not a hauling-logic bug and not HD-reachable. Mentioned only to note LWM uses fragile transpilers on TryFindBestBetterStoreCellFor / HaulToCellStorageJob / ListerHaulablesTick; HD composes via postfix/prefix on the same methods, so if an LWM transpiler fails to apply on a future RW patch, HD's behavior degrades to vanilla cell selection (still safe), not a crash.
- ⚠️ applies to HD — **PlaceHauledThingInCell storing-delay toil 'could break the entire chain' when a pawn moves a lot of things at once (PUAH-style)** _(LWM Deep Storage Deep_Storage_Pause.cs:114-117 author comment; About 'Haul To Stack? The residual is extra churn/wasted trips when HD's no-cell-reservation races multiple pawns onto the same delayed DSU cell. Verify in-game; bounded, never loss.
- n/a to HD — **Over-capacity check disabled (hard) for Project RimFactory / PRF-lite / Extended Storage to prevent pawns getting stuck in a relocation loop** _(LWM Deep Storage Patch_TryFindBestBetterStoreCellFor_ShouldRemoveFrom.cs:51-63 Prepare())_ — LWM disables its own over-capacity eviction when another storage mod can't expose CompDeepStorage, because the two storage systems fight in a relocation loop. Lesson reinforced: don't relocate items OUT of a foreign storage mod's cells — HD's StorageRouting only relocates loose/lower-priority floor stacks toward storage, gated by WorthRelocating + priority eligibility.
- ⚠️ applies to HD — **Storage Network StartJob auto-spawn infinite-loop guard (Force-Wait after 3 throttle hits/tick)** _(Storage Network Patch_Pawn_JobTracker_StartJob_AutoSpawn.cs:50-53,73-89 (ForceWaitThreshold=3, ForceWaitTicks=60))_ — SN itself had to add a per-tick throttle + Force-Wait because repeated same-tick materialization attempts caused vanilla job loops. But if HD's job re-issues / re-plans repeatedly in a tick (work-scan churn), it could. The opt-in default-off shim limits exposure. Verify under the opt-in with a large network manifest.

### General-pattern recommendations
- VALIDATE-THROUGH-VANILLA (HD's stated and verified design): every candidate cell is gated by StoreUtility.IsGoodStoreCell (→ NoStorageBlockersIn, patched by all three mods + LWM's per-carrier intelligence postfix), CanStackWith, parent.Accepts, and space is priced via GetItemStackSpaceLeftFor / GetMaxItemsAllowedInCell. This makes modded acceptance, per-cell multi-stack capacity, and carrier rules apply automatically for ANY Building_Storage-derived mod — no per-mod patch. Confirmed in HaulToStack.FindStackCell, BulkHaul.StorageSpaceForDef, StorageRouting.FindBeforeCarryStoreCell, unload FindTargetOrDrop.
- DEPOSIT-RE-GATE IS AUTHORITY: HD's plan-time space/availability estimates are deliberately conservative upper bounds; the actual put-away (FindTargetOrDrop re-runs TryFindBestBetterStorageFor per carried stack; Toils_Haul.PlaceHauledThingInCell re-runs IsGoodStoreCell on arrival) deposits only what each modded cell accepts and floor-drops/re-routes the remainder. This bounds every over-estimate or race to one-cycle churn, never overfill, never cargo loss, never storage-rule bypass — the single most important defense across this family.
- TREAT NON-SPAWNED CONTAINER CONTENTS AS OUT-OF-REACH: HD's sweep/route/stack/count features draw only from spawned floor stacks (listerHaulables / ThingsOfDef filtered to Spawned). Spawned-model mods (ASF, LWM) are seen and handled; virtual/despawned-model mods (Storage Network) gracefully no-op (their items are materialized by the mod's own on-demand spawn at job start, not pre-counted by HD). HD's only network-aware code is an explicit, reflection-only, opt-in, default-off shim — never a hard dependency.
- HD MOD-SPECIFIC PATCHES ARE NARROW AND COMPOSITIONAL: HD touches the shared storage-search methods only via Patch_StorageFilter_Funnel (postfix that ONLY vetoes/narrows __result in non-default contexts, byte-inert otherwise) and HaulToStack (postfix that refines the CELL within vanilla's already-chosen room/priority, re-gated by IsGoodStoreCell). Because HD only narrows (never expands or invents a cell), it composes safely with ASF/LWM/SN transpilers/prefixes/postfixes on the same methods regardless of Harmony order.
- NEVER EVICT FROM A FOREIGN STORAGE MOD'S CELLS: HD relocates only loose / lower-priority floor stacks toward better storage (StorageRouting.WorthRelocating + priority eligibility); it never pulls items OUT of a storage building to relocate them. This avoids the two-storage-system relocation loop LWM itself had to hard-disable for PRF/Extended Storage.

### Adversarial verification

## Family: `inventory-keepers`

HD already has a mature, well-architected keep-item system that severs the unload<->refetch loop for Simple Sidearms, Combat Extended, Smart Medicine, Dub's Bad Hygiene and Yayo's Combat 3. The single shared choke point is InventorySurplus.SurplusOf (C:/Users/Arthur/Sync/Projects/RimWorld Mods/HaulersDream/Source/HaulersDream/InventorySurplus.cs:31), which returns 0 ("keep the whole stack") for any item IsManagedKeepItem (line 244) recognizes, and KeepCountOf (line 290) honors the three vanilla keep sources (drug-policy takeToInventory, inventoryStock, packable food) plus the CE loadout reserve. The risk surface is SMALL because: (a) the only feature that can strip a foreign keep-item is the "unload all surplus" option (unloadAllSurplus, default FALSE — HaulersDreamSettings.cs:36); with it OFF, AdoptSurplusInventory only adopts player-ruled defs (PawnUnloadChecker.cs:244), so keeper mods are safe by default; (b) HD never uses vanilla JobGiver_DropUnusedInventory (which is where SS/SM/ItemPolicy install their protective patches), it uses its own JobDriver_UnloadHauledInventory, which is precisely WHY a per-mod bridge is needed; (c) HD's tagging self-heal and scoop pick already exclude Simple Sidearms remembered sidearms (YieldRouter.InventoryStackOfDef:573, CompHauledToInventory self-heal:132). The ONE genuinely unprotected mod is ITEM POLICY (new, no shim): it keeps an arbitrary count of ANY ThingDef per pawn via its own Dictionary (ItemPolicyUtility.policies), re-fetches via JobGiver_TakeItemForInventoryStock (a TakeInventory job when pawn.inventory.Count(def) < N), and only protects DRUGS, and only against vanilla DropUnused, via a ShouldKeepDrugInInventory postfix — so HD's surplus unload (when unloadAllSurplus=true) WILL strip ItemPolicy-stocked items (drug and non-drug alike) and trigger the loop. The best GENERAL fix is two-fold: (1) replace HD's inline drug-keep mirror with an actual call to vanilla JobGiver_DropUnusedInventory.ShouldKeepDrugInInventory(pawn, thing) for drug items, which automatically honors EVERY keeper mod's postfix on that method (covers Smart Medicine's stocked drugs AND ItemPolicy's stocked drugs in one hook, replacing/backstopping the SM reflection shim for drugs); and (2) add a general "inventory-stock contract" reader that, in addition to vanilla inventoryStock, reflects ItemPolicy's per-pawn GetItemPolicyEntry(pawn, def) keep-count — covering ItemPolicy's NON-drug items (which have no vanilla hook). ItemPolicy is best handled as a small targeted reflection shim feeding the general KeepCountOf path, because its policy data is its OWN dictionary, not vanilla inventoryStock, and no general vanilla signal exposes it.

## Family: `crafting-bills`

HD's batch-craft / bill-routing subsystems overlap heavily with this family but are, with one real exception, already well-guarded. The single high-risk item is GENERAL, not mod-specific: HD's full-replace prefix on BillRepeatModeUtility.MakeConfigFloatMenu (returns false) hides the repeat-mode entries of ANY mod that transpiles that method. HD fixed this for Everybody Gets One via EverybodyGetsOneCompat.TryInsertModes, but the bridge is EGO-only — any other mod that injects modes the same way is still hidden. Recommended general fix: instead of returning false, let vanilla (and thus every foreign transpiler) build the base list, then ADD HD's batch entries via a postfix (or call all foreign inserters generically). Two medium items: (a) Cook Carefully's poison-discard transpiles Toils_Recipe.FinishRecipeAndStartStoringProduct, which HD's JobDriver_BatchCraft REPLICATES (does not call) — so a batched CookMealCarefully (it is allowMixingIngredients=true and now batchable) banks poisoned meals instead of discarding them; CanBatch should exclude recipes another mod special-cases at the finish step (targeted exclusion of CookMealCarefully is the practical fix). (b) HD and Common Sense BOTH transpile WorkGiver_DoBill.TryFindBestBillIngredientsInSet_AllowMix's single SortBy call with no Harmony priority, so which spoiling-sort wins is load-order dependent (no crash; both do spoiling-first, so cosmetic). The DoBill ingredient-gather loop with Common Sense is already correctly ceded via CommonSenseCompat.OwnsDoBillFlow (CS defaults make it own the flow). HD's banked-inventory CountProducts postfix is verified correct against decompiled Verse.RecipeWorkerCounter (vanilla counts inventory only when includeEquipped) and is deliberately mode-agnostic so it also fixes EGO's count-based modes. CanBatch correctly excludes EGO's dynamic person-count modes. The mech "10 jobs/tick" loop (a bug Common Sense itself fixed in 1.3.4) is guarded by BillRouteGate.WorkerMayShareCraft.

### General-pattern recommendations
- Menu/option-list patches must COMPOSE, not REPLACE: when HD augments a vanilla UI list-builder (MakeConfigFloatMenu), prefer a Postfix that appends to / re-opens the menu, OR transpile the same insertion point, so every other mod's transpiler on that method still runs. A full-replace prefix that returns false is a monopoly that hides all peers and forces a per-mod bridge (the EGO TryInsertModes is exactly that debt). The general rule: never `return false` from a prefix on a method other mods are known to transpile for additive UI.
- CanBatch (and any HD path that re-implements a vanilla recipe finish) must exclude recipes another mod special-cases at the finish/product step. HD already excludes UFT, take-entire-stacks, non-Bill_Production, and non-vanilla repeat modes; add 'recipes whose finish is transpiled by another mod' (detected by soft-dep recipe-defName match where a generic detector is impossible). General principle: if HD replaces a vanilla method instead of calling it, HD inherits responsibility for every mod that patched the original — either route through the real method or exclude the special-cased inputs.
- Cede to any mod that OWNS a shared driver flow, keyed as generically as feasible: HD's CommonSenseCompat.OwnsDoBillFlow is the right pattern (detect a foreign rewrite of JobDriver_DoBill.MakeNewToils and stand down all conversions). Consider generalizing the detection from 'CS settings fields' to 'a non-HD Prefix is registered on JobDriver_DoBill.MakeNewToils that returns false' so a future similar mod is also handled.
- When two mods transpile the SAME single instruction (here the AllowMix SortBy), set an explicit Harmony priority or make the matcher robust to a prior wrapper, so behavior is load-order-independent. At minimum, match structurally (HD does) and tolerate the case where the target call was already replaced (degrade to no-op, log once).
- Count-correction at the SOURCE (RecipeWorkerCounter.CountProducts postfix) is the right general place to make HD-banked inventory visible to ANY count-based repeat mode (vanilla TargetCount AND EGO's modes), gated on !includeEquipped to avoid double-count — verified against decompiled Verse.RecipeWorkerCounter. This is a model for 'fix it once where every consumer reads it' rather than per-mode patches.
- Mech/non-colonist worker exclusion (BillRouteGate.WorkerMayShareCraft) is the general guard against the '10 jobs/tick' loop class that affects ANY DoBill-rerouting feature; keep it the single source of truth across all bill-conversion entry points (HD already does).

### General-pattern recommendations
- Generic-base @__instance guard: any Harmony patch on a method inherited from a generic base with reference-type instantiations MUST guard the body with `TargetType.IsInstanceOfType(__instance)` (and take `object __instance`) so it doesn't fire for sibling instantiations sharing the JIT native code. Reusable for ANY mod that derives multiple WorkGiver/JobDriver/etc. from one generic base.
- Reflection-only, degrade-SAFE soft-dep shim: a single *Compat class (no `using <ForeignMod>`, no foreign type in any signature — everything held as Thing/object), lazy Init, cached IsActive that is true ONLY when every load-bearing member binds, each member null-guards + fails OPEN, exactly one permitted fail-open swallow for a pure-redundancy call. Covers ANY optional dependency.
- Skip-embarked-holder via the foreign mod's own 'is-contained' predicate (InVehicle) in per-pawn enumeration loops, while letting PARKED/normal holders pass the existing liveness+CurJob eligibility unchanged — so the desirable case (eat-from/build-from a parked vehicle's cargo) keeps working and only the unreachable case is skipped. Covers any mod whose carrier type subclasses Pawn.
- Deposit through the foreign mod's event-correct API (AddOrTransfer) — not a raw container move — when depositing into a foreign carrier, so the foreign mod's events/manifest stay consistent; clamp the deposit to the SINGLE matched transferable using the foreign mod's OWN matcher, never a def-sum. Covers any mod with a pending-load manifest decremented inside its add API.
- Defense-in-depth carrier-type skip ([UC1]/[UC2]) gated on the foreign-type check ONLY (not a feature toggle), so a safety/correctness guard fixing a pre-existing misfire can never be switched back off by a feature flag. Covers any mod whose carrier type HD's generic carrier-handling must never claim.
- Fail-on-despawn + salvage-not-drop for a disappearing deposit target (vehicle launch/despawn): FailOnDespawnedOrNull + re-validate spawned + finish-action releases claims and re-tags the carried split into inventory. Covers any mod with a deposit target that can vanish mid-job.
